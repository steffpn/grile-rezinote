1
Create Supabase Project & Collect Keys
  1. Go to supabase.com and create a new project.
  2. Choose region EU West (Ireland) or EU Central (Frankfurt) for best latency from Romania.
  3. Set a database password — use only letters and numbers (no special characters like ! @ # to avoid URL-encoding issues).
  4. Wait ~2 minutes for provisioning to complete.
Password tip: Use a simple alphanumeric password like MySecurePass2026abc. Special characters in the password break the connection string URL format and cause "Failed to retrieve database credentials" errors.

Collect These Values

Save all of these — you'll need them for Vercel environment variables.

VariableWhere to Find
NEXT_PUBLIC_SUPABASE_URLSettings > API > Project URL
NEXT_PUBLIC_SUPABASE_ANON_KEYSettings > API > anon public key
SUPABASE_SERVICE_ROLE_KEYSettings > API > service_role secret key
DATABASE_URLSettings > Database > Connection string > URI tab > Transaction pooler (port 6543)
DATABASE_URL format:
postgresql://postgres.YOURPROJECT:PASSWORD@aws-0-eu-central-1.pooler.supabase.com:6543/postgres

Use the Transaction pooler (port 6543), not the direct connection (port 5432). The app is configured for transaction pooling.
2
Push Database Schema

This creates all 12 database tables in your Supabase project. Run this from the project root on your local machine:

Windows (PowerShell)

$env:DATABASE_URL="postgresql://postgres.YOURPROJECT:PASSWORD@aws-0-eu-central-1.pooler.supabase.com:6543/postgres"; pnpm db:push

macOS / Linux (Bash)

DATABASE_URL="postgresql://postgres.YOURPROJECT:PASSWORD@aws-0-eu-central-1.pooler.supabase.com:6543/postgres" pnpm db:push

Replace the connection string with your actual DATABASE_URL from Step 1.

Expected output: Drizzle Kit will show all tables being created (users, chapters, questions, options, attempts, attempt_answers, subscriptions, specialties, admission_data, audit_logs, webhook_events, site_settings) and several enums.

Seed Default Settings

Go to Supabase Dashboard > SQL Editor and run:

INSERT INTO site_settings (key, value, updated_at) VALUES ('exam_duration_seconds', '14400', now()) ON CONFLICT (key) DO NOTHING;

This sets the default exam duration to 4 hours (14400 seconds).

3
Configure Supabase Auth & Database Trigger

3A. Auth Settings

In the Supabase dashboard:

  1. Authentication > Providers > Email — make sure it's enabled (it is by default).
  2. Authentication > URL Configuration:
    • Site URL: https://your-vercel-domain.vercel.app (set this after Step 5 when you know your Vercel URL, or use a placeholder for now)
    • Redirect URLs: Add https://your-vercel-domain.vercel.app/auth/confirm
    • Also add http://localhost:3000/auth/confirm for local development
  3. Authentication > Email Templates — the defaults are fine. Just verify that the Confirm and Recovery templates contain {{ .ConfirmationURL }}.

3B. Create Database Trigger

CRITICAL: Without this trigger, signup will appear to succeed but users will be stuck in an infinite redirect loop. The app requires a row in public.users for every authenticated user, and this trigger creates it automatically on signup.

Go to SQL Editor and run this entire block:

-- Function: create a public.users row when a new auth user signs up CREATE OR REPLACE FUNCTION public.handle_new_user() RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER SET search_path = '' AS $$ BEGIN INSERT INTO public.users (id, email, full_name, year_of_study, role, is_superadmin) VALUES ( NEW.id, NEW.email, COALESCE(NEW.raw_user_meta_data->>'full_name', 'Utilizator'), (NEW.raw_user_meta_data->>'year_of_study')::integer, 'student', false ); RETURN NEW; END; $$; -- Trigger: fire after every new auth user insert CREATE OR REPLACE TRIGGER on_auth_user_created AFTER INSERT ON auth.users FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();

3C. Enable Row-Level Security

This prevents users from reading each other's data through the Supabase API. Run in SQL Editor:

-- Enable RLS on all tables ALTER TABLE public.users ENABLE ROW LEVEL SECURITY; ALTER TABLE public.subscriptions ENABLE ROW LEVEL SECURITY; ALTER TABLE public.chapters ENABLE ROW LEVEL SECURITY; ALTER TABLE public.questions ENABLE ROW LEVEL SECURITY; ALTER TABLE public.options ENABLE ROW LEVEL SECURITY; ALTER TABLE public.attempts ENABLE ROW LEVEL SECURITY; ALTER TABLE public.attempt_answers ENABLE ROW LEVEL SECURITY; ALTER TABLE public.specialties ENABLE ROW LEVEL SECURITY; ALTER TABLE public.admission_data ENABLE ROW LEVEL SECURITY; ALTER TABLE public.site_settings ENABLE ROW LEVEL SECURITY; ALTER TABLE public.audit_logs ENABLE ROW LEVEL SECURITY; ALTER TABLE public.webhook_events ENABLE ROW LEVEL SECURITY; -- Users: can read own row CREATE POLICY "Users can read own profile" ON public.users FOR SELECT USING (auth.uid() = id); -- Subscriptions: can read own row CREATE POLICY "Users can read own subscription" ON public.subscriptions FOR SELECT USING (auth.uid() = user_id); -- Chapters: all authenticated users can read CREATE POLICY "Authenticated users can read chapters" ON public.chapters FOR SELECT USING (auth.role() = 'authenticated'); -- Questions: all authenticated users can read CREATE POLICY "Authenticated users can read questions" ON public.questions FOR SELECT USING (auth.role() = 'authenticated'); -- Options: all authenticated users can read CREATE POLICY "Authenticated users can read options" ON public.options FOR SELECT USING (auth.role() = 'authenticated'); -- Attempts: can read own attempts CREATE POLICY "Users can read own attempts" ON public.attempts FOR SELECT USING (auth.uid() = user_id); -- Attempt answers: can read own answers CREATE POLICY "Users can read own answers" ON public.attempt_answers FOR SELECT USING ( attempt_id IN ( SELECT id FROM public.attempts WHERE user_id = auth.uid() ) ); -- Specialties: all authenticated can read CREATE POLICY "Authenticated users can read specialties" ON public.specialties FOR SELECT USING (auth.role() = 'authenticated'); -- Admission data: all authenticated can read CREATE POLICY "Authenticated users can read admission data" ON public.admission_data FOR SELECT USING (auth.role() = 'authenticated'); -- Site settings: all authenticated can read CREATE POLICY "Authenticated users can read site settings" ON public.site_settings FOR SELECT USING (auth.role() = 'authenticated');
Note: Server-side code (Server Actions, API routes) connects via DATABASE_URL using the Postgres driver directly, which bypasses RLS. These policies only affect the Supabase JS client used in middleware for session-aware queries.
4
Set Up Stripe Products & Webhook

4A. Create Products

Go to dashboard.stripe.com. Make sure you're in Test Mode (toggle in top-right) for initial setup.

  1. Go to Products > + Add product
  2. Product 1: Abonament Lunar (Monthly)
    • Name: Abonament Lunar
    • Pricing model: Recurring
    • Amount: 49 RON / month
    • Save → copy the Price ID (starts with price_...)
    • This is your STRIPE_MONTHLY_PRICE_ID
  3. Product 2: Abonament Anual (Annual)
    • Name: Abonament Anual
    • Pricing model: Recurring
    • Amount: 396 RON / year (33 RON/month)
    • Save → copy the Price ID
    • This is your STRIPE_ANNUAL_PRICE_ID

4B. Collect API Keys

Go to Developers > API keys:

VariableWhere
STRIPE_SECRET_KEYSecret key (starts with sk_test_)
STRIPE_PUBLISHABLE_KEYPublishable key (starts with pk_test_)

4C. Create Webhook Endpoint

Do this after Step 5 — you need your Vercel URL first. Come back to this step after deploying.
  1. Go to Developers > Webhooks > + Add endpoint
  2. Endpoint URL: https://your-vercel-domain.vercel.app/api/webhooks/stripe
  3. Click "Select events" and check these 5 events:
    • customer.subscription.created
    • customer.subscription.updated
    • customer.subscription.deleted
    • invoice.payment_succeeded
    • invoice.payment_failed
  4. Click "Add endpoint"
  5. Click into the endpoint → Reveal the Signing secret
  6. Copy it (starts with whsec_...) → this is your STRIPE_WEBHOOK_SECRET
5
Deploy to Vercel

5A. Push Code to GitHub

Make sure your code is on the main branch and pushed:

git push origin main

5B. Import to Vercel

  1. Go to vercel.com > Add New Project
  2. Import your GitHub repository
  3. Framework: Next.js (auto-detected)
  4. Before clicking Deploy, set up environment variables (next step)

5C. Set Environment Variables

In the Vercel import screen (or Settings > Environment Variables after import), add all 11 variables:

Chicken-and-egg: For NEXT_PUBLIC_SITE_URL, NEXT_PUBLIC_APP_URL, and STRIPE_WEBHOOK_SECRET — you need the Vercel URL first. Use a placeholder for the first deploy, then update after you have the URL. Set STRIPE_WEBHOOK_SECRET to a dummy value like whsec_placeholder for now.

5D. Deploy

Click Deploy. Wait for the build to complete (~2-3 minutes).

Note your production URL, e.g. https://grile-rezinote.vercel.app

6
Connect Everything (URLs & Webhook)

Now that you have your Vercel URL, wire everything together:

6A. Update Vercel Environment Variables

Go to Vercel > Project Settings > Environment Variables and update:

  • NEXT_PUBLIC_SITE_URL → https://your-app.vercel.app
  • NEXT_PUBLIC_APP_URL → https://your-app.vercel.app

6B. Create Stripe Webhook

Go back to Step 4C and create the webhook endpoint now that you have your URL.

Then update the Vercel env var:

  • STRIPE_WEBHOOK_SECRET → whsec_... (the real signing secret)

6C. Update Supabase Auth URLs

Go to Supabase > Authentication > URL Configuration:

  • Site URL: https://your-app.vercel.app
  • Redirect URLs: Add https://your-app.vercel.app/auth/confirm

6D. Redeploy

Go to Vercel > Deployments → click ... on the latest deployment → Redeploy.

This picks up the updated environment variables.

After redeploy: Your app should be live and functional at your Vercel URL.
7
Create Your Superadmin Account
There is no admin registration UI. You sign up as a normal user, then promote yourself via SQL. This is intentional for security.
  1. Go to https://your-app.vercel.app/signup
  2. Create an account with your email. Fill in name and year of study (any value is fine for admin).
  3. Check your email for the verification link. Click it to verify.
  4. Open Supabase Dashboard > SQL Editor and run:
    UPDATE users SET is_superadmin = true WHERE email = 'your-email@example.com';

    Replace your-email@example.com with the exact email you signed up with.

  5. Refresh the app or navigate to /admin. You should see the admin panel.

Admin Panel Routes

URLPurpose
/adminAdmin dashboard
/admin/chaptersCreate, edit, reorder, delete chapters
/admin/questionsManage questions (CS/CM types, source references)
/admin/import-exportBulk import/export questions from CSV or Excel
/admin/specialtiesManage dental specialties

Troubleshooting

  • Redirected to /dashboard instead of /admin? — The email in your SQL didn't match. Run SELECT email FROM users; in SQL Editor to see the exact email stored.
  • Stuck on "Se creeaza contul..."? — The database trigger might not be set up. Go back to Step 3B.
  • No verification email? — Check Supabase > Authentication > Logs. If rate-limited, wait an hour or temporarily disable email confirmation in Providers > Email.
8
Verify Everything Works

Run through this checklist to confirm the deployment is functional:

Authentication

  • Sign up with a new email → receive verification email
  • Click verification link → redirected to /dashboard
  • Log out → redirected to /login
  • Log in again → back to /dashboard
  • Forgot password → receive reset email → set new password

Admin Panel

  • Superadmin can access /admin
  • Regular user cannot access /admin (redirected to dashboard)
  • Create a chapter
  • Create a CS question and a CM question
  • Import questions from a CSV file

Practice & Exams

  • Start a chapter practice test → answer → see results
  • Start a mixed practice test
  • Start an exam simulation → timer counting down
  • Close browser and return → answers preserved

Payments (Stripe Test Mode)

  • Visit /pricing → see monthly and annual plans
  • Click subscribe → Stripe Checkout opens
  • Use test card 4242 4242 4242 4242 (any expiry, any CVC)
  • Payment succeeds → redirected to /checkout/success
  • /subscription shows active subscription

Dashboard

  • After a test, dashboard shows updated stats
  • Charts render (trend chart, radar chart)
  • Answer history shows completed answers

PWA (Mobile)

  • On mobile, browser shows "Add to Home Screen"
  • After installing, app opens without browser chrome
9
Production Hardening (Before Real Users)

Go-Live Checklist

  • Switch Stripe to live keys (sk_live_, pk_live_)
  • Create live Stripe products/prices (test ones don't work in live mode)
  • Create a live webhook endpoint with the production URL
  • Update all Vercel env vars with live Stripe keys and webhook secret
  • Re-enable email confirmation in Supabase Auth if you disabled it
  • Set up a custom domain in Vercel (optional but professional)
  • Update Supabase Site URL and Redirect URLs to custom domain
  • Redeploy after all env var changes

Security Fixes (from audit)

3 Critical Issues found by security audit. Fix these before allowing real users:
#IssueImpactFix
1 SQL injection in practice.ts Attacker could read/modify database Replace sql.raw() with sql.join() in the wrongAnswersOnly code path (~line 144)
2 No rate limiting on auth Brute-force login attacks possible Add @upstash/ratelimit or Vercel WAF rate limiting to login/signup/forgot-password
3 Missing security headers Clickjacking, MIME sniffing Add X-Frame-Options, CSP, HSTS headers in next.config.ts

Full audit report at .planning/security/audit-report.md in the repository.
