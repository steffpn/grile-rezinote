1. Executive Summary

grile-ReziNOTE is a SaaS platform purpose-built for Romanian dental students preparing for the national residency entrance exam. The platform was designed, implemented, and verified across 10 development phases, delivering a feature-complete v1.0 product in approximately 7 hours of execution time.

Core differentiator delivered: After completing a simulated exam, students instantly see whether their score would have earned admission to specific dental specialties based on real historical threshold data from the past 5 years. No Romanian competitor offers this feature.

What Was Built

  • Exam simulation engine matching the real format: 200 questions (50 CS + 150 CM), timed with server-authoritative countdown, using the official Romanian scoring methodology
  • Practice quiz system with chapter-based and mixed modes, immediate or deferred feedback, and a wrong-answer review system with mastery tracking
  • Admission comparison engine that maps simulation scores against 5 years of historical admission thresholds per specialty
  • Admin content management with chapter CRUD, question CRUD (CS/CM types), and bulk CSV/Excel import-export with Romanian diacritics support
  • Analytics dashboard with per-chapter stats, trend charts, radar visualization, heatmap, sparklines, and answer history
  • Peer comparison system with anonymous leaderboard, percentile ranking, score distribution, and contextual motivational messages
  • Stripe subscription billing with monthly/annual plans in RON, hosted checkout, webhook-driven sync, trial system, and paywall
  • Progressive Web App installable from any mobile browser with custom install prompt, bottom tab bar, 44px touch targets, and offline indicator
  • Full authentication with email/password registration, email verification, password reset, and persistent sessions
  • Branded landing page with 7 sections communicating platform value in Romanian

4. Architecture

System Overview

Clean Next.js monolith with three-layer separation: App Router (client), Server Actions + Route Handlers (API), and Supabase PostgreSQL (data). No microservices, no separate backend.

+---------------------------------------------------------------------+ | CLIENT LAYER (PWA) | | +-----------+ +-----------+ +-----------+ +------------------+ | | | Landing / | | Quiz | | Dashboard | | Admin Panel | | | | Auth | | Engine | | Analytics | | Content Mgmt | | | +-----------+ +-----------+ +-----------+ +------------------+ | +------|----------------|----------------|----------------|------------+ | | | | +------v----------------v----------------v----------------v------------+ | API LAYER (Server Actions + Route Handlers) | | +-----------+ +-----------+ +-----------+ +------------------+ | | | Auth | | Quiz | | Analytics | | Admin / Stripe | | | | Actions | | Actions | | Actions | | Actions | | +------|----------------|----------------|----------------|------------+ | | | | +------v----------------v----------------v----------------v------------+ | DATA LAYER (Supabase PostgreSQL + Drizzle ORM) | | +-------+ +----------+ +---------+ +--------+ +--------+ +------+ | | | users | | chapters | |questions| |attempts | |subscrip| |admis.| | | +-------+ +----------+ +---------+ +--------+ +--------+ +------+ | +------|---------------------------------------------------------------| | | +------v---------------------------------------------------------------+ | EXTERNAL SERVICES | | +------------------+ +------------------+ +-------------------+ | | | Stripe | | Supabase Auth | | Serwist (PWA) | | | | Webhooks+Billing | | Email+Sessions | | Service Worker | | | +------------------+ +------------------+ +-------------------+ | +----------------------------------------------------------------------+

Key Architectural Patterns

PatternDescriptionLocation
Question Bank / Engine Separation Admin CRUD (bank) is fully decoupled from quiz-taking logic (engine). Questions are snapshotted at attempt creation time. lib/actions/practice.ts, lib/actions/exam.ts
Server-Authoritative Timer Server records startedAt + timeLimit. Client derives remaining time from server timestamps. Server rejects submissions past deadline + grace period. lib/actions/exam.ts
Scoring as Pure Function Stateless function takes questions + answers, returns score breakdown. Tested with 23 unit tests covering all CM annulment edge cases. lib/quiz/scoring.ts
Webhook-Driven Billing Stripe webhooks drive subscription state changes. Event ID deduplication prevents double-processing. Database is updated transactionally. api/webhooks/stripe/route.ts, lib/stripe/webhook-handlers.ts
Immediate vs Deferred Feedback feedbackMode stored on attempt record. API conditionally reveals correct answers based on mode. Exam always uses deferred. lib/actions/practice.ts
Answer Snapshotting attempt_answers.isCorrect set at scoring time, never recalculated. Protects historical scores from retroactive question edits. lib/db/schema.ts

8. Key Decisions

Significant architectural and product decisions made during development, with rationale.

Roadmap

Scoring engine built in Phase 1, not Phase 4/5. Every downstream feature (practice tests, exam simulation, analytics, peer comparison, admission comparison) depends on correct scoring. Building and testing it first with 23 unit tests prevented cascading bugs.

Phase 1

Next.js 15 chosen over 16. Version 16 deprecated middleware patterns needed for auth. Version 15 has wider ecosystem compatibility with Serwist and other packages. Clear upgrade path available.

Phase 1

CM scoring: max 950 points (50 CS x 4 + 150 CM x 5). Confirmed against official Romanian methodology. CM annulment: 0 points for fewer than 2 or more than 4 selections, validated by 23 passing tests covering all edge cases.

Phase 3

Server-side superadmin check in layout.tsx, not middleware. Middleware runs on every request and is best kept minimal. Layout-level getCurrentAdmin() provides defense with simpler logic. All admin Server Actions independently call getCurrentAdmin().

Phase 3

Soft delete with archivedAt timestamps. Questions and chapters are soft-deleted (not hard-deleted) to preserve historical attempt data integrity. Answer snapshots reference original question IDs.

Phase 3

Client-side CSV/Excel parsing. Avoids uploading large files to the server. PapaParse and ExcelJS run in the browser. UTF-8 BOM prepended to CSV exports for correct Romanian diacritics rendering in Excel.

Phase 6

Majority-of-years rule for admission comparison. A student is shown as "admitted" to a specialty if their score exceeds the threshold in the majority (3+ of 5) of historical years. Provides realistic assessment accounting for year-to-year variation.

Phase 9

7-day trial starts on first feature access, not at signup. Users who register but do not immediately use the platform do not waste their trial period. Trial tracked server-side via users.trialStartedAt.

Phase 10

Split icon entries (any/maskable separate) for PWA. Broader install compatibility across different device manufacturers. start_url set to /dashboard since authenticated users land there.

Phase 10

44px minimum touch targets enforced across all interactive elements. Applied to quiz option buttons, exam navigator, chapter selector, CTA buttons, and practice config form. Matches Apple HIG and WCAG 2.5.5 guidelines.
